# 网站 Beta 发布工作计划

本文件用于整理当前网站从 Alpha 原型推进到可对外发布 Beta 版本所需完成的工作，作为接下来逐项推进的统一基线。

说明：

- 本文件聚焦 `网站 Beta 发布前工作`，不替代 `docs/website/website_design_spec.md` 的产品设计说明。
- 本文件默认建立在当前网站 Alpha 已完成的基础上。
- 本文件不对任务做优先级分层；以下各项均视为 Beta 发布前需要完成的正式工作。
- 自本计划建立后，结果解释、正式人物素材、新验证体系与真源治理已完成一轮落地；后续应重点关注仍未完成的剩余收尾项。

---

## 一、Beta 目标

当前 Beta 阶段的目标不是继续搭建页面骨架，而是把现有网站推进到以下状态：

- 结果页具备足够的解释力与说服力
- 网站侧匹配逻辑与当前项目真源保持可复核的一致性
- 图片与展示素材达到正式发布所需的基本规范
- 全站文案、交互与视觉完成度达到可对外展示标准
- 前端工程结构与文档状态适合继续维护与迭代

---

## 二、Beta 发布前需要完成的工作

### 1. 结果解释层

目标：

- 让结果页真正回答“为什么像这个人”
- 让综合结果、风格结果、理念结果都具备可读解释

需要完成的工作：

- 设计结果解释的数据结构
- 从匹配结果中提取关键命中维度与差异维度
- 为 `综合最像` 生成结构化解释文案
- 为 `风格最像` 与 `理念最像` 生成分层解释或解释摘要
- 让解释文案与实际匹配逻辑保持一致
- 清理当前仍带占位性质的结果页说明

交付结果：

- 结果页具备真实解释，而不是占位解释

### 2. 匹配真源与网站侧一致性治理

目标：

- 确保网站里的匹配逻辑持续对齐当前项目真源

需要完成的工作：

- 重新审查 `matching_engine.py` 与兼容层入口当前承担的职责
- 梳理网站侧 `matching` 模块与 Python 真源的逐项对应关系
- 复核：
  - 单题相似度
  - 维度聚合
  - 稀有维度权重
  - 核心 / 参考加权
  - 核心惩罚
  - 签名维度惩罚
- 确认网站侧不存在“看起来一样、实际上漂移”的实现差异
- 视情况决定是否进一步收拢匹配引擎真源命名与边界

交付结果：

- 网站匹配逻辑与项目真源的一致性基线清楚可查

### 3. 新验证体系

目标：

- 建立一套替代旧测试体系的 Beta 校验机制

需要完成的工作：

- 建立网站侧与 Python 侧的一致性校验
- 做若干领导人画像自喂回归
- 做关键人物簇边界样本验证
- 校验五类题型编码与匹配输入的一致性
- 做一轮完整端到端流程回归：
  - 欢迎页
  - 引导页
  - 第一部分
  - 第二部分
  - 结果页
  - 重新测试
- 形成可重复执行的最小验证入口

交付结果：

- 一套新的 Beta 验证基线

### 4. 正式人物素材治理

目标：

- 让结果页图片资产达到可对外发布的基本标准

需要完成的工作：

- 审查 17 位人物当前图片是否都适合正式展示
- 替换不理想的图片
- 统一裁切、焦点位置与展示规范
- 建立图片来源记录：
  - 文件名
  - 来源链接
  - attribution
  - 许可协议
- 决定页面内或文档中的署名呈现方式
- 对敏感人物展示方式做一次产品语境审查

交付结果：

- 一套正式的人物图像资产与来源记录

### 5. 结果页展示收尾

目标：

- 把结果页从“接近完成”推进到“可对外展示”

需要完成的工作：

- 审查主结果第一屏的信息层级
- 审查人物图像、姓名、角色、摘要之间的权重关系
- 审查“为什么像 / 仍有差异”的解释区是否平衡
- 审查风格 / 理念卡片是否清楚但不过度抢主结果
- 审查补充结果区是否保留、压缩或重写
- 决定是否展示分值、展示到什么程度、以什么语气展示

交付结果：

- 一版稳定、克制且完整的 Beta 结果页

### 6. 欢迎页 / 引导页 / 章节页收尾

目标：

- 让进入结果之前的整段体验形成完整、统一的节奏

需要完成的工作：

- 审查欢迎页文案与结构是否成立
- 审查引导页是否还存在多余解释或提示
- 审查第一部分 / 第二部分章节页是否节奏统一
- 审查欢迎页、引导页、章节页之间的视觉与语气是否一套
- 对必要位置做小幅修正，而不是大重构

交付结果：

- 更统一的前段体验与页面节奏

### 7. 答题体验 QA

目标：

- 确保答题链路在真实使用场景下稳定、顺畅、可理解

需要完成的工作：

- 抽查五类题型在桌面端和手机端的表现
- 重点检查：
  - 双极量表两端是否清楚
  - 点数分配题是否稳定
  - 长题干是否挤压选项
  - 自动推进是否顺畅
  - 回改是否清楚
  - 最后一题进入结果是否明确
- 做一轮完整人工体验流程记录
- 修掉影响真实使用的细节问题

交付结果：

- 一版经答题体验 QA 收口的 Beta 链路

### 8. 全站文案一致性收口

目标：

- 让所有页面都用同一种产品语气说话

需要完成的工作：

- 检查：
  - 欢迎页
  - 引导页
  - 章节页
  - 答题页提示语
  - 结果页文案
  - 免责声明
  - 图片 attribution
  - 按钮文案
- 统一表达原则：
  - 不夸大准确性
  - 不写成政治判定器
  - 不写成知识测试
  - 不使用过多防御性句式
  - 不保留多余系统感提示语

交付结果：

- 一套一致、克制、自然的全站文案口径

### 9. 前端工程整理

目标：

- 让当前网站代码结构适合继续演进，而不是停留在 Alpha 快速实现状态

需要完成的工作：

- 拆分 `App.tsx`
- 抽离页面组件：
  - 欢迎页
  - 引导页
  - 章节页
  - 答题页
  - 结果页
- 抽离流程状态逻辑
- 抽离结果整理层
- 抽离解释生成层
- 梳理 `website/src` 内的数据、组件、结果、匹配模块边界
- 清掉明显的 Alpha 快速实现痕迹

交付结果：

- 一套更清晰、可维护的 Beta 前端结构

### 10. 发布前工程收口

目标：

- 确保网站具备稳定构建与对外访问条件

需要完成的工作：

- 确认 `npm run generate:data` 与 `npm run build` 稳定
- 检查生成产物与源数据关系是否清楚
- 检查 dev / build 行为是否一致
- 检查本地化图片资产、来源记录与部署行为是否稳定
- 检查 tunnel / host / 域名访问是否正常
- 明确 Beta 对外访问方式
- 做一次发布前 sanity check

交付结果：

- 一套可以支撑 Beta 展示与发布的工程运行基线

### 11. 项目状态与文档同步

目标：

- 让仓库文档与 Beta 实现状态重新保持一致

需要完成的工作：

- 更新 `now.md`
- 更新 `project_overview.md`
- 更新 `current_effective_files.md`
- 更新 `docs/website/website_design_spec.md`
- 新增或更新 Beta 状态说明文件
- 明确写清：
  - Beta 已完成内容
  - 当前仍有限制
  - 当前验证到什么程度
  - 当前结果应如何表述

交付结果：

- 一套与 Beta 实现同步的项目文档

### 12. Beta 发布说明与交付边界

目标：

- 明确 Beta 版本对外应该怎么说、说到什么程度

需要完成的工作：

- 定义 Beta 对外口径
- 定义 Beta 当前已知限制
- 定义 Beta 后的下一阶段目标
- 形成一份阶段说明，作为后续发布说明基础

交付结果：

- 一份清楚的 Beta 阶段说明基线

---

## 三、执行方式

接下来建议采用逐项推进方式：

- 一次只推进一个工作包
- 每完成一个工作包后，再进入下一项
- 若某个工作包内部需要继续拆分，再另行细化其实施方案

---

1. 结果页展示收尾
2. 欢迎页 / 引导页 / 章节页收尾
3. 答题体验 QA
4. 全站文案一致性收口
5. 前端工程整理
6. 发布前工程收口
7. 项目状态与文档同步
8. Beta 发布说明与交付边界
