# 网站 Alpha 实施工作计划

本文件用于记录网站 Alpha 阶段的正式实施计划，作为接下来开发推进、阶段汇报与自检的统一依据。

说明：

- 本文件聚焦 `网站 Alpha 实现流程`，不替代 `website_design_spec.md` 的产品设计说明。
- 本文件默认建立在当前已确认方案之上：
  - 网站放在 `website/` 子目录
  - 技术栈采用 `React + Vite + TypeScript`
  - 第一版先做纯前端、本地数据驱动
  - 网站使用专用结构化题库文件
  - 匹配逻辑封装为独立 `matching` 模块
  - 结果页先使用统一占位图，不等待正式照片资产
  - 风格/理念切换使用独立轻量过渡页
  - 点数分配题采用 `+/-` 卡片交互
  - 先冻结关键文案，其余允许占位
  - Alpha 目标是 `全流程可用版`

---

## 一、Alpha 目标

Alpha 阶段的完成标准是：

- 四段式流程完整可跑：`欢迎 -> 引导 -> 答题 -> 结果`
- `82` 题全部接入网站
- `5` 类题型全部可用
- 网站侧匹配引擎可运行，并能输出：
  - `综合最像`
  - `风格最像`
  - `理念最像`
- 桌面端与手机端均可正常完成作答流程
- 页面具备明确的 `社论档案感` 视觉方向

Alpha 阶段暂不要求：

- 正式人物照片库
- 真实匹配理由生成
- 后端 API
- 账号系统与中途保存
- Top 3 深度展开
- 人物详情页

---

## 二、总执行顺序

Alpha 实施按以下顺序推进：

1. 工程基线
2. 类型与 schema
3. 题库结构化
4. 人物展示数据层
5. 匹配引擎移植
6. 应用状态与流程状态机
7. 页面骨架
8. 答题页容器
9. 五类题型组件
10. 结果页实现
11. 视觉系统与响应式
12. 自检与回归
13. Alpha 收尾文档

---

## 三、分阶段实施计划

### 阶段 1：工程基线

目标：把 `website/` 从已安装依赖的目录，整理为真正可开发的前端工程。

任务：

- 补齐 `vite` 入口与配置
- 补齐 `TypeScript` 配置
- 建立基础目录结构：
  - `src/app`
  - `src/pages`
  - `src/components`
  - `src/features/questions`
  - `src/features/matching`
  - `src/features/results`
  - `src/features/profiles`
  - `src/data`
  - `src/styles`
  - `src/lib`
- 建立全局样式入口与基础 design tokens
- 保证本地开发环境可启动

阶段完成标志：

- 网站工程可以正常启动
- 空白应用壳可渲染
- 目录职责清晰

### 阶段 2：类型与 schema

目标：先把网站内部的数据结构与答案编码约定全部定死。

任务：

- 定义题库 schema
- 定义答案编码 schema
- 定义匹配结果类型
- 定义人物展示类型
- 定义页面与流程状态类型
- 固定五类题型答案编码：
  - 单端 Likert：`1-7`
  - 双端 Likert：`1-7`
  - 情境单选：字母值
  - 强制权衡：字母值
  - 点数分配：对象值

阶段完成标志：

- TypeScript 类型系统可供整个网站复用
- 不再需要在页面组件内临时决定数据结构

### 阶段 3：题库结构化

目标：将 `final_questions.md` 中的 `82` 题转换为前端直接可消费的数据文件。

任务：

- 逐题录入网站专用题库数据
- 为每题补齐至少以下字段：
  - `id`
  - `displayNo`
  - `layer`
  - `section`
  - `dimensionId`
  - `type`
  - `prompt`
  - `description`
  - `leftLabel/rightLabel`
  - `options`
  - `maxPoints`
- 与 `item_dimension_mapping.json`、`item_id_mapping.md` 校对顺序和编号
- 标记风格向理念切换点

阶段完成标志：

- 网站拥有完整结构化题库
- 页面层可以只消费数据，不再依赖 Markdown 文件

### 阶段 4：人物展示数据层

目标：把“匹配计算数据”和“结果展示数据”分开。

任务：

- 提取网站结果页所需人物展示字段：
  - `leader_id`
  - 中文名
  - slug
  - 占位图 key
  - 简短占位介绍
- 建立统一占位图策略
- 为未来正式照片与来源记录预留字段

阶段完成标志：

- 结果页可独立读取安全、统一的展示数据

### 阶段 5：匹配引擎移植

目标：将当前 Python 真源等价移植为网站侧 `matching` 模块。

任务：

- 提炼并移植：
  - `itemSimilarity`
  - `blockMultiplier`
  - `dimWeight`
  - `layerScore`
  - `rankAnswers`
- 接入：
  - `leader_profiles`
  - `item_dimension_mapping.json`
  - `signature_dimensions.json`
- 封装统一入口，供 UI 侧直接调用
- 做最小烟测，避免明显行为漂移

阶段完成标志：

- 网站侧可以根据答案直接生成排序结果
- 页面层无需理解具体评分细节

### 阶段 6：应用状态与流程状态机

目标：先把四段式流程控制逻辑搭好。

任务：

- 定义流程节点：
  - `welcome`
  - `intro`
  - `question`
  - `transition`
  - `result`
- 定义答题状态：
  - 当前题 index
  - 已答题 map
  - 是否首次作答
  - 是否完成全部题目
  - 是否显示理念过渡页
  - 是否允许进入结果
- 定义重测逻辑
- 定义最后一题手动结算逻辑

阶段完成标志：

- 页面状态流转规则明确
- 自动推进与回改逻辑有统一状态基础

### 阶段 7：页面骨架

目标：先把四段式页面壳子搭出来。

任务：

- 搭建欢迎页
- 搭建引导页
- 搭建答题页
- 搭建结果页
- 搭建风格/理念过渡页

阶段完成标志：

- 全流程可点击进入各页面
- 页面结构和信息层级完整出现

### 阶段 8：答题页容器

目标：让单题答题体验先成立。

任务：

- 实现题号显示与进度条
- 实现题目头部与说明区域
- 实现上一题 / 下一题 / 查看结果按钮逻辑
- 处理首次作答自动推进
- 处理回改不自动推进
- 处理最后一题特殊结算按钮
- 标识当前题与已答状态

阶段完成标志：

- 答题页框架可稳定切题
- 只差题型组件即可形成完整答题流程

### 阶段 9：五类题型组件

目标：完成 Alpha 最大工作量。

任务：

- 实现单端 Likert 组件
- 实现双端 Likert 组件
- 实现情境单选组件
- 实现强制权衡组件
- 实现点数分配组件
- 每个组件支持：
  - 已答回显
  - 修改答案
  - 首次作答自动推进钩子
  - 手机端可点击热区
- 点数分配题额外支持：
  - 总分必须为 `10`
  - 剩余分数明确显示
  - 未分配完成时不视为有效作答

阶段完成标志：

- `82` 题全部可以真实作答
- `5` 类题型全部可用

### 阶段 10：结果页实现

目标：形成网站 Alpha 的结果闭环。

任务：

- 调用匹配引擎生成结果
- 提取：
  - `综合最像`
  - `风格最像`
  - `理念最像`
- 展示主结果：
  - 姓名
  - 占位图
  - 占位匹配理由
- 展示次结果：
  - 风格最像：姓名
  - 理念最像：姓名
- 加免责声明与重新测试按钮

阶段完成标志：

- 用户可以完整走到结果页
- 结果页主次层级清晰

### 阶段 11：视觉系统与响应式

目标：把网站从“能用”提升到“像这个项目”。

任务：

- 建立颜色、排版、间距、边框、阴影变量
- 落地 `社论档案感` 视觉方向：
  - 旧纸米白
  - 炭黑正文
  - 深墨绿 / 海军蓝强调
  - 少量暗金属点缀
  - 标题衬线、正文黑体
- 欢迎页与结果页加入轻量背景氛围
- 答题页保持克制与高可读性
- 优化手机端：
  - 按钮热区
  - 双端量表不拥挤
  - 底部操作区稳定

阶段完成标志：

- Alpha 视觉风格明确
- 桌面端与手机端都具备基本完成度

### 阶段 12：自检与回归

目标：在当前没有正式测试体系的情况下，完成网站 Alpha 最小可信自检。

任务：

- 跑构建
- 检查 TypeScript / Vite 错误
- 抽测五类题型
- 完整走通一次主流程：
  - 欢迎页
  - 引导页
  - 风格题
  - 过渡页
  - 理念题
  - 结果页
  - 重新测试
- 至少做一次“人物画像自喂”的烟测
- 检查关键移动端视图

阶段完成标志：

- 网站 Alpha 基本可用
- 已知问题被整理为待修清单

### 阶段 13：Alpha 收尾文档

目标：为后续 Beta 迭代保留清晰边界。

任务：

- 记录 Alpha 已完成项
- 记录 Alpha 暂未完成项：
  - 正式照片库
  - 真实匹配理由
  - 更细结果层
  - 后端/API
- 记录 Beta 候选项
- 如有必要，回写 `website_design_spec.md` 的实现状态说明

阶段完成标志：

- Alpha 形成可交接、可继续推进的明确基线

## 五、当前关键风险点

- `82` 题结构化录入工作量较大，是 Alpha 的硬成本
- 点数分配题是五类题型中最容易拖慢实现进度的一类
- 匹配引擎移植必须严格对齐当前真源，不能按摘要文档另写近似实现
- 当前尚无正式照片资产，结果页只能先使用统一占位图
- `run_matching_tests.py` 文件名仍会误导，移植时必须按“匹配引擎真源”理解，而不是按“测试脚本”理解

## 六、当前执行起点

Alpha 开工时，优先从以下三项开始：

1. 整理 `website/` 工程基础配置
2. 定义题库 schema、答案编码和 TypeScript 类型
3. 开始将 `82` 题转换为前端结构化题库数据

这三项完成后，再进入匹配引擎移植与页面实现阶段。
